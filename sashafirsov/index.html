Responsive layout

html body

For html and body padding and margins are eliminated as those would be responsibility of immediate content: header, nav, main

Vertical flex layout on body level defined top, main, and toolbar zones. Header resides on top, toolbar in mobile mode sits on bottom, leaving main zone scrollable.

Main section scrolling

Achieved by flex-grow: 100; overflow: auto; for main content;

Toolbar position

in default (mobile) layout it is set on the bottom for one-hand operations convenience. For same reason the most frequent operations like "schedule" in sample are set closer to right side. Apologies to left-handed folks.

header

To make H1 text scale down the font-size is set relative to viewport width.
The initial value 6vw is set to make font size growing till the moment the height matches default browser one: 2em or xx-large.
The default font size and view-port width relative equality defines the breakpoint when @media(min-width: 34.35rem) (550px in 100% zoom) would set the H1 font size to default one.
Two parameters above are critical to a) keep increasing the font not more than default while expanding the width, b) keep header from changing the height when text become 2-liner on narrow view or font size increased in wide view.
Those numbers would need adjustment to accommodate the header text length change.

menu

Menu is triggered by restyled checkbox.
For browser without support of checkbox restyling the LABEL tag would hover and mimic checkbox UI.

onchange checkbox event handler triggers search-gr-on header class which changes search group visibility.

search

mobile behavior
By default (mobile) the search UI is hidden behind of icon button( same as in menu ). Once triggered, search group expands and takes over whole header space. The trigger icon changes to on the left, the center is taken by search input field, and "search" button on the right.
Since trigger button stays same, SPACE multiple times would trigger search group on and off. To get into input field use TAB (as expected).

Desktop behavior
Enabled by "large mobile screen" breakpoint. Search group in this mode comprise the input field and button.

unfocused input filed is translucent and takes small width just barely sufficient for hint text. On focus input became opaque and expands, covering the menu space. Menu is kept 3 chars wide to keeping ability to use menu and unfocus the search.

nav

Serves as primary actions toolbar along with menu floating over page content.

In Desktop mode navigation shifted beneath header.

main

main content is surrounded by header and actions bar. It has own scroll area which replaces the page scrolling. flex css styling helps to stretch main content to view area remaining after header and action bar.

Responsive breakpoints

Overriding the browser font side on page is a disrespectful to user and breaks the accessibility of page. User could change the default page font size according to own vision or situation. It could be also changed dynamically reflecting say user preferences in portrait/landscape mode or connecting external screen like TV.

Since the letter size could vary from user to user, the pixel defined breakpoints do not reflect the visual design which in most of web applications rely on text dimensions.

How to define true responsive breakpoints?

By visual design
Usually mockups are created for targeted devices which reflect as physical dimensions as conditions the user observing and interacts with UI.
Sometimes mockup is created per UX component instead of page which created. It is natural to use component-aware and component-specific breakpoints. Which could lead to inconsistent across page and applications series of breakpoints. Leading to user confusion, additional effort of test automation, customer support issues, etc.
It is up to product to enforce media breakpoints across applications/pages/components.
By reducing the number of media breakpoints app will reduce the number of UX variations at expense of per-component UX polishing.
REM match to pixel
"classic" pixel-perfect approach operates in pixels and responsive breakpoints usually defined same way. To define same size in REM units, the simplest way is to set two sibling DOM nodes and assign the width in PX and REM accordingly. In given page 45rem breakpoint matches 720px taken in Desktop Chrome.

Icons

Even simplified image in order to be clearly recognized has to be at least 2x larger than than letter symbols next to it with exception of large title text.
That is why icon font size here is set to 2rem

Accessibility

Keyboard navigation is supported by DOM order of focus-able components.

Focus UX should be distinctive and do not hide the outline. In this page it is done by radical pink color and visually destructive inset shadow with 1 second transition animation.
The shadow is used instead of border color to
a) avoid collision with styles of navigation components which often use borders ( i.e. button links )

Frameworks with component set always have own way of making focused elements visually and dynamically exposed.

Unadvised

order-dependent selectors for groups visibility switch
DETAILS/SUMMARY for groups visibility switch, Edge browser does not support it.
checkbox as group visibility switch, Edge browser does not support UI customization. Work around(used here) exist on HTML+CSS level.

Use LABEL for visibility group change. I.e. do not use LABEL for focus to input field if it leads to components group visibility change like in case of search input. It prevents to focus on triggering element and breaks the keyboard navigation. Instead, use A link with in-page navigation.
